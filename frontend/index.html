01 — Overview

The Problem

Airbnb hosts in Jamaica — particularly in tourism-heavy markets like Montego Bay and Negril — frequently misprice their listings. Overpricing leads to low occupancy; underpricing leaves revenue on the table. A data-driven price recommendation system addresses this directly.

The model accepts nine property-level inputs and returns a predicted nightly rate in USD, along with model metadata and inference latency. The architecture is intentionally extensible: swapping the underlying model, adding features, or retargeting a different market requires touching a single file.

02 — Architecture

Request Flow

POST /api/v1/predict └─ InputValidator range/type checks → clean 422 on bad input └─ AmenityScoreTransformer weighted composite amenity feature └─ InteractionFeatures capacity_index, bath_ratio, guest_density └─ LocationEncoder aligned OHE — no train/serve skew └─ StandardScaler └─ XGBoost → predicted price └─ JSON response with latency_ms

All preprocessing stages are composed into a single sklearn Pipeline and serialised together with the model. There is no separate preprocessing script — the same transformation logic runs at training time and inference time, by construction.

03 — Feature Engineering

Beyond Raw Features

The dataset provides nine inputs. Four engineered features are added via custom sklearn-compatible transformers before the model sees any data:

FeatureFormulaRationale
amenity_scoreWeighted sum of amenity flagsPool (2×) and A/C (1.5×) command premiums in Jamaica's climate; composite avoids adding 5 sparse dimensions
capacity_indexaccommodates ÷ bedroomsSpace efficiency — a 1-bed sleeps-6 villa is fundamentally different from a 1-bed studio
bath_ratiobathrooms ÷ bedroomsLuxury indicator — en-suite ratios correlate with premium positioning
guest_densityaccommodates × amenity_scoreDemand proxy × quality proxy — non-linear interaction the model would otherwise miss
04 — Model Selection

05 — API Design

Production Flask REST API

Built with Flask using the application factory pattern — the app instance is never in module scope, which makes testing and multiple configurations straightforward. Routes are versioned blueprints under /api/v1/.

Endpoints

  • POST /api/v1/predict — single listing, full validation, returns price + latency metadata
  • POST /api/v1/predict/batch — up to 100 listings per request
  • GET /api/v1/health — liveness probe for Docker / load balancer health checks
  • GET /api/v1/model-info — exposes training metadata, CV results, feature importances

Key Engineering Decisions

  • Model caching: pipeline is loaded once via @lru_cache(maxsize=1) — zero I/O overhead after cold start
  • Input validation: InputValidator raises ValueError on bad inputs, caught at route level, returned as 422
  • Structured logging: every prediction logs location, bedrooms, price, and latency_ms at INFO; every validation error at WARNING
  • Error taxonomy: 400 (bad JSON), 422 (validation), 503 (model not ready), 500 (unexpected) — all return JSON

06 — Deployment

Docker + EC2

The service uses a two-stage Docker build. The builder stage installs dependencies in isolation; the runtime stage copies only what's needed into a minimal python:3.11-slim image.

  • Non-root execution: container runs as a dedicated appuser — no process runs as root
  • Health check: Docker polls /api/v1/health every 30s; 3 failures mark the container unhealthy
  • Production server: gunicorn with 2 workers — Flask's dev server never runs in production
  • Model at build time: training runs during docker build — the image ships with a ready artefact, no separate seed step needed
docker-compose up --build

08 — What's Next

Further Development

If I were taking this to production at scale:

  • Real data: replace synthetic dataset with scraped or licensed listings — synthetic R² doesn't transfer to production accuracy
  • Experiment tracking: MLflow or W&B to persist every training run and make model selection reproducible
  • SHAP explanations: /api/v1/explain endpoint returning per-feature SHAP values — hosts see which features drive their price
  • CI/CD: GitHub Actions to run tests, build Docker image, push to ECR on every merge to main
  • Drift monitoring: track prediction distribution over time to detect market shifts
